##Difference between OSI and TCP/IP?
  Answer:OSI model has 7 layers, TCP/IP has 4 layers.
  The TCP /IP Model is the foundational framework that dictates how data is sent and received over internet. It organizes the network communication into four        abstraction layers. Application, Transport, Internet and Network access layers ensuring seemless data exchange between different systems and hardwares.
  #Application Layer: The  top most layer allows user facing software  and application to interact with the network. It provides standardized services so programs   can understand the data being sent . HTTP/HTTPS, FTP,SMTP,DNS and Telnet protocols
  #Transport Layer:This layer ensures reliable and orderly data delivery between devices. It breaks data down into segements and handles error checking, flow        control and re transmission of lost packets. TCP or UDP Protocols
  #Internet Layer:Responsible for logical addressing and routing, this layer packages data into packages and routes to the best path to send across multiple         networks. IP,ICMP ARP Protocols
  #Network Access Layer:The base layer handles the physical transmission of data accross the newtwork hardwares.(Ethernet cables, wifi routers) It formats data      into frames and uses physical hardware (MAC) address to route data to the next immediate device. Ethernet, wi-fi, token ring.
  TCP/IP combines OSI's Application, presentation nand session layer into single Application Layer. and merges OSI's Physical and Dta and Link layer into Network    access Layer. OSI is generally used as a generic reference and teachjing tool, While TCP/IP is the specific implementation used to connect hosts over the          internet.

##What is TCP Hansshake?
  Answer:The TCP handshake (often called a 3-way handshake) is a three-step process used by the Transmission Control Protocol to create a reliable, connection-oriented link between a client and a server before any actual data is transferred. It ensures both devices are synchronized and ready to communicate.
  Step 1: SYN (Synchronize)The client initiates the connection by sending a TCP packet with the SYN flag set. This packet tells the server that the client wants to start a communication session and specifies the client's initial sequence number.
  Step 2: SYN-ACK (Synchronize-Acknowledgment)The server receives the client's SYN and replies with a packet that has both the SYN and ACK flags set. This confirms receipt of the client's request (Acknowledgment) and simultaneously signals the server's own sequence number, indicating it is ready to communicate.
  Step 3: ACK (Acknowledgment)The client receives the server's SYN-ACK packet and replies with a final ACK packet. This acknowledges the server's sequence number.
  It's important because The handshake is essential for maintaining the robustness of the internet. Because TCP requires confirmation at every step, it ensures ordered and error-free data delivery. It prevents lost packets, handles flow control, and protects against duplicate or corrupted data transmissions.
##What happens if SYN Packet is lost?
  Answer:If a TCP SYN packet is lost, the connection initiation pauses. The client's operating system triggers a retransmission timer (usually starting at 1 second) and will repeatedly resend the SYN packet until it receives a SYN-ACK or hits a maximum timeout limit, at which point the connection attempt will fail.

##DNS generally usees TCP OR UDP?
  Answer:DNS relies on both TCP and UDP, assigning specific roles to each protocol. UDP handles routine name lookups because it is lightweight and fast, while TCP handles larger queries (like DNSSEC or IPv6 records), zone transfers, and encrypted traffic like DNS over TLS (DoT).
##Why doesnot Youtube use only TCP for streaming Gaming videos?
  Answer:YouTube relies on UDP (specifically via the QUIC protocol) alongside TCP for gaming and live streams to eliminate delay.For gaming, speed and real-time interaction are more important than avoiding a few minor visual glitches.

##What happens after SYN-ACK?
  Answer:After SYN-ACK, the client sends an Acknowledgment (ACK) packet back to the server. This completes the standard TCP 3-way handshake. Once the server receives the ACK, the connection status changes to ESTABLISHED on both sides, and actual data transfer (like requesting a webpage) can begin.
